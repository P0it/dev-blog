# /research — 개념 → 다중소스 조사 → 종합 글 (직접 경로)

조사할 개념 / 주제 / 앵글: **$ARGUMENTS**

URL 하나를 번역·요약하는 `/draft` 와 달리, 이 커맨드는 **여러 소스를 조사해
하나의 글로 종합**한다. 출처를 그대로 옮기는 게 아니라 **여러 자료를 녹여
작성자 자신의 개념으로 재구성**한다. `ai_jobs` 큐·워커·`claude` CLI 를 거치지
않고 이 세션이 직접 쓴다.

## `/draft` 와 다른 점 (먼저 숙지)

- **소스가 여럿**이다 — 단일 원문이 아니라 검색으로 찾은 여러 글·문서·저장소를 취합.
- **고정 구조에 매이지 않는다** — "개념→사례→생태계" 같은 순서 강제 없음.
  모은 내용을 작성자 관점에서 가장 자연스러운 흐름으로 재구성한다.
- **시각자료는 "공식 이미지 우선, 없으면 생략"** — 카탈로그 카드를 글마다
  기본으로 깔지 않는다(매 글 같은 카드는 자동생성 티가 난다). 아래 "시각자료
  방침" 을 따른다.
- **참고 자료는 공식 이미지를 쓸 때만 단다** — 이미지를 가져오면 `## 참고 자료`
  로 원출처를 밝힌다. 이미지를 안 쓰면 참고 자료·저작권 줄도 넣지 않는다(여러
  소스를 녹여 작성자 개념으로 쓴 글이므로).
- 그 외 **말투·제목·헤드라인 규약은 POSTING.md 를 그대로** 따른다.

## 시각자료 방침 (★ 핵심) — 혼합: 공식 우선 + SVG 채움

글마다 반복되는 **카탈로그 카드(```visual)는 쓰지 않는다**(자동생성 티). 대신
**소제목(`##`)마다** 그 맥락에 어울리는 시각자료를 하나씩 넣되, 아래 우선순위로:

1. **공식 이미지 우선.** 그 소제목 내용에 맞는 공식 문서·원본의 실제 다이어그램·
   도표가 있으면 그걸 쓴다.
   - **자체 버킷에 재호스팅**(핫링크 금지): `npm run fetch:image -- <원본 이미지 URL>`
     → 출력 공개 URL 을 `![설명](그 URL)` 으로 박는다.
   - 쓰기 전 `curl -sIL <URL>` 로 200·image/* 인지 확인하면 헛수고를 던다.
2. **공식 이미지가 없으면 직접 SVG 일러스트를 그린다**(```illustration). POSTING.md
   "```illustration" 절의 하우스 스타일을 반드시 따른다 — `viewBox="0 0 800 450"`,
   stroke-width 2.5, 색은 디자인 토큰(`var(--diag-blue-fill)` 등)으로, 3색 이내,
   첫 자식 `<title>`. 라인아트는 워커 없이 본문에 바로 인라인 렌더된다
   (`markdownRender.tsx` → `Illustration`). 맥락에 맞는 도식을 그때그때 그린다.
   - 디자인 토큰: `--diag-{blue,purple,teal,green,yellow,red,mute}-{fill,stroke}`,
     `--diag-edge`, `--diag-cluster-bg`, `--fg-strong`, `--fg-neutral`.
3. 카드보다 차라리 한 줄 문장이 나은 소제목(마무리 감상 등)은 **굳이 그림을 넣지
   않아도 된다** — 억지 채우기 금지.
4. **공식 이미지를 하나라도 썼으면 `## 참고 자료`** 에 그 원출처(이미지가 실린
   페이지·라이선스)를 적는다. 직접 그린 SVG 만 있으면 참고 자료는 불필요하다.
5. 외부 AI 래스터 이미지 생성(DALL·E 등)은 아직 미설치 — 쓰려면 별도 셋업 필요.

## 절차

1. **규약 숙지** — `POSTING.md` 의 "2-0. 제목"(후킹+SEO), "2. 말투·본문 골격",
   "2-1. 강조", "2-2. 시각자료 카탈로그" 를 읽는다. [[korean-writing-style]] 준수.

2. **다중소스 조사** — 주제를 여러 각도로 검색한다. 깊은 조사가 필요하면
   `deep-research` 스킬을 써도 된다(팬아웃 검색·소스 페치·사실 검증·종합).
   가볍게는 `WebSearch` 로 후보를 찾고 `WebFetch` 로 핵심 소스를 읽는다.
   - **개념** 자체: 공식 문서·대표 소개 글로 정확히 잡는다.
   - **실제 outcome·사례**: 그 기술을 써서 나온 결과(예: 토큰 절감, 성능 개선)를
     다룬 공개 글·벤치마크·케이스 스터디.
   - **생태계**: 그 기술로 만들어진 도구·저장소·Claude skill 등.
   - 주제에 따라 어떤 갈래가 중심인지는 달라진다 — $ARGUMENTS 의 앵글을 따른다.

3. **사실 검증** — 종합 글이라 출처를 본문에 밝히지 않더라도, **수치·고유명사·
   인용은 실제 소스에서 확인된 것만 쓴다**. 토큰 절감률·벤치마크 점수 같은
   숫자를 지어내지 않는다(POSTING.md "사실은 정확하게"). 소스마다 수치가
   엇갈리면 보수적으로 쓰거나 범위로 적는다. 감상·해석은 자유롭게 얹되 사실과
   구분되게.

4. **종합·집필** — 모은 내용을 **작성자의 개념으로 재구성**해 한국어 글 1편을 쓴다.
   - 본문은 `> 요약`(두괄식 훅·excerpt) → 서술형 문어체 `##` 헤드라인 순서.
     맨 위에 `# 제목` 을 다시 쓰지 않는다.
   - 순서·구조는 자유 — 가장 잘 읽히는 흐름으로. 한 헤드라인 = 한 메시지,
     그 아래는 가볍게(분량과 호흡).
   - `-습니다`체 + 부드러운 종결(`~죠`/`~까요?`) 양념, 번역체·매문장 출처표딱지 금지.
   - **시각자료는 위 "시각자료 방침"** 대로 — 소제목마다 공식 이미지(재호스팅) 우선,
     없으면 하우스 스타일 SVG(```illustration)를 직접 그린다. 카탈로그 카드는 안 쓴다.
   - **공식 이미지를 하나라도 넣었으면** 끝에 `## 참고 자료` 로 원출처를 적는다.
     직접 그린 SVG 만 있으면 참고 자료 없이 마지막 헤드라인(정리·시사점 또는
     작성자의 마무리 감상)으로 맺는다.

5. **파일 작성** — `drafts/<slug>.md` 를 만든다(`drafts/` 는 .gitignore 됨).
   ```
   ---
   title: 후킹+검색어를 담은 한국어 제목
   tags: [태그A, 태그B]
   category: ai            # 선택 — tech/business/design/claude/infra/ai/insights 중
   ---
   > 요약 인용구…
   ## 헤드라인…
   ```
   `source_url` 은 단일 출처가 없으므로 넣지 않는다.

6. **적재** — `npm run draft -- push drafts/<slug>.md` 실행.

7. 출력된 `/admin/editor?slug=…` 경로를 사용자에게 알린다. 검토 후 에디터에서 발행한다.

형식·옵션 상세: `docs/draft-in-claude-code.md`. 단일 URL 글은 `/draft`.
