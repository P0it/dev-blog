# 포스팅 작성 가이드 — Anthropic 글 번역·요약

주 워크플로우: [anthropic.com](https://www.anthropic.com/) 의 post / story / announcement / research →
**번역 → 요약 → 본인 해석** 후 발행.

> 코드 변경 없는 **작성 컨벤션** 문서다.
> 출처/참고자료는 별도 기능이 아니라 본문 markdown 안에 약속된 형식으로 적는다.
> (`Post` 타입의 본문은 `bodyMd` 단일 필드 — `src/lib/types.ts`,
> 렌더링은 `src/components/post/MarkdownView.tsx`, `react-markdown + remark-gfm`)

## 1. 에디터 입력 — velog식 미니멀

`/admin/editor`(`src/components/admin/PostEditor.tsx`)는 velog처럼 **제목·태그·카테고리·본문** 넷만 입력한다.
나머지는 전부 자동이다.

| 필드 | 어떻게 채워지나 |
|---|---|
| 제목 | 상단 큰 입력칸에 직접. 직역·점잖은 제목 ✗ — 클릭하고 싶게 후킹하는 제목으로 (2-0절) |
| 태그 | 제목 아래 입력칸에 쉼표 구분 (`anthropic, 보안, ai-security` 등) |
| 카테고리 | 작은 드롭다운 (`anthropic` 계열 / 주제 카테고리) |
| 본문 | 마크다운. 이미지는 **드래그·붙여넣기**로 자동 업로드(별도 버튼 없음) |
| ~~슬러그~~ | **자동** — 제목에서 slugify(ASCII 영문/숫자/하이픈만, 한글·기호 제거). 기존 글은 URL 안정성을 위해 원래 슬러그 유지. 한국어 제목은 영문 자동 fallback(`post-<timestamp>`) 으로 떨어지니 인물명·핵심 키워드 슬러그를 쓰려면 `/draft` 직접 경로에서 frontmatter `slug:` 에 명시한다 |
| ~~요약(excerpt)~~ | **자동** — 본문 첫 `>` 블록쿼트에서 파생(`deriveExcerpt`). 카드/검색/SNS/SEO/RSS/OG가 보는 문장과 페이지상 두괄식 훅이 **같은 문장**이다 |
| ~~썸네일 종류~~ | **자동** — 슬러그 해시로 a–f 6변형에 결정적 분포(`thumbKindFromSlug`). 카드 단조로움 방지 |
| ~~읽는 시간~~ | **자동** — 본문 분량에서 산출(`deriveReadingMin`) |
| ~~시리즈 · 추천 옵션~~ | 입력란 없음. 기존 값은 row update에 포함 안 해 보존됨 |

> 본문은 `> 요약` → `## 헤드라인` 순서로 시작한다.
> 맨 위에 `# 제목`을 다시 쓰지 않는다 — 제목은 입력칸이 권위.
> 본문 첫 `>` 인용구는 페이지의 두괄식 훅이면서 동시에 `excerpt` 컬럼에 자동 추출되어 카드·OG·RSS에 동일 문장으로 들어간다.

## 2-0. 제목 — 핵심 내용을 요약한 후킹

글 제목(에디터 상단 입력칸)은 그 콘텐츠의 **핵심 내용을 한 줄로 요약**해, 독자가 클릭하고 싶게 만드는 문구다.
이 블로그에서 제목은 곧 `<title>` 태그이자 H1·OG 제목이라 그 페이지 SEO·클릭률의 가장 큰 지렛대다.
직역식·밋밋한 제목 ✗.

**인물·회사명을 제목에 앞세우지 않는다.**

- 인터뷰 대상자·발표자·회사 이름은 제목의 주인공이 아니다.
  그 정보는 **`##` 소제목**이나 **발표자 소개 문단**에 두면 충분하다("발표자 소개" 절).
- 제목은 "누가 말했나"가 아니라 **"무슨 내용인가"**를 담는다.
  독자는 사람 이름보다 그 사람이 한 말의 알맹이에 끌려 클릭한다.
- ✗ `gstack — YC 사장 Garry Tan이 Claude Code를 엔지니어링 팀으로 굴리는 법`
  ○ `Claude Code 하나로 23명짜리 엔지니어링 팀을 굴린다` (인물·소속은 본문으로)
- ✗ `Claude Code 창시자 Boris Cherny — "코딩은 이미 끝났다, 다음은 무엇인가"`
  ○ `"코딩은 이미 끝났다" — 손코딩 0, 하루 PR 150개의 현재`
- 예외: 그 인물·회사 자체가 독자가 검색할 핵심어일 만큼 주제와 한 몸일 때만 제목에 남긴다.

**핵심 내용을 요약한다.**

- 그 글에서 가장 강한 한 가지(수치·반전·결론)를 제목에 끌어온다.
- 숫자·의외의 사실이 들어가면 더 잘 걸리고 더 잘 눌린다.
- 제목만으로 본문의 알맹이가 가늠되게 — 단, 결론을 다 까발리지 않고 궁금증은 남긴다.
- 위트·반전의 한 마디로 클릭을 부른다 — 내용 없는 낚시 ✗.

**검색어도 함께 고려한다.**

- 독자가 칠 법한 키워드는 **인물 이름이 아니라 내용 키워드**(주제어·사건·제품·개념)다.
  소스 유출 사건을 다룬다면 사람들은 "Claude Code 유출"을 친다 — 그 말을 제목에 박는다.
- 핵심 키워드는 앞쪽에. 검색 결과에서 제목은 잘린다(한글 30자 안팎).
- 같은 키워드를 excerpt(첫 인용구)·`##` 헤드라인에도 자연스럽게 반복.
- 제목만은 본문 말투 규약에서 자유.

예: `Claude Code 소스 유출 사태, Anthropic의 대응이 의외였다`
(검색어 "Claude Code 소스 유출" + 후킹 "대응이 의외였다" — 인물명 없이 사건·결론으로)

> 제목만으로 검색 상단에 오르지는 않는다.
> 사이트 신뢰도·콘텐츠 깊이·경쟁 강도가 함께 작용한다.
> 경쟁이 덜한 구체적 키워드(롱테일)일수록 작은 블로그도 걸릴 여지가 크다.

## 2. 말투 · 본문 골격

**목적: 요약하고, 재미있게 풀어 쓴다.**
이 블로그는 단순 정보 제공처가 아니라 **개인 기록 블로그**다.
원문을 통째로 옮기지 않는다 — 한 번 소화한 뒤 핵심만 추려, 내 말로 다시 들려준다.

**사실은 정확하게.**
수치·고유명사·발언(인용) 같은 사실관계는 원문 그대로 둔다.
사실을 지어내거나 왜곡하지 않는다.
인용은 발화자를 원문대로 밝힌다.

**감상·해석은 자유롭게.**
사실을 옮기는 데 그치지 않고, 글쓴이의 반응·해석·평가를 자연스럽게 섞어도 된다.
"의외였다", "이래서 흥미롭다" 같은 개인적인 시선이 곧 개인 블로그의 색이다.
단, 사실과 감상은 독자가 구분할 수 있게 쓴다 — 감상을 사실인 것처럼 끼워 넣지 않는다.

**말투: `-습니다` 뼈대에, 대화형 종결을 양념으로.**
이 블로그의 톤 모델은 보도자료가 아니라 **Claude Chat·ChatGPT·Gemini의 답변체**다.
사실은 단단하게, 말투는 사람이 옆에서 설명하듯 — 그러면서도 **글의 무게는 지킨다**.
사건·공지·기술 글이 일상 블로그처럼 가벼워지지 않게 한다.

- **뼈대는 `-습니다 / -입니다`.**
  문장의 대부분은 격식 종결이다. 핵심 사실·수치·결론은 항상 격식으로 못 박는다.
- **양념으로 섞는 종결.**
  - `~죠 / ~겠죠` — 가볍게 동의를 구하거나 짚어 줄 때. ("원인은 단순하죠.")
  - `~까요? / ~일까요?` — 질문을 던져 독자를 끌어들일 때.
  - `~네요` — 가벼운 반응. 드물게.
  - `~거든요` — 배경·이유를 한 박자 풀어 줄 때. 한 글에 1~2회 선에서.
  - `~다고 합니다 / ~라고 합니다` — 출처 있는 사실을 부드럽게 전할 때.
- **피하는 종결.**
  - `~해요 / ~예요 / ~잖아요 / ~더라고요` 같은 본격 해요체 — 글의 무게를 떨어뜨린다.
  - `~한 셈입니다 / ~기에 충분합니다 / ~에 다름 아닙니다` 같은 번역서 종결.

비율 감각은 단순하다 — `~습니다` 대 양념 종결은 대략 **8 : 2 ~ 7 : 3**.
양념이 한 문단 걸러 깔리면 느끼해진다.
단정할 곳은 `~습니다`로 단정하고, 흐름이 굳어 보이는 곳만 풀어 준다.
신문 보도체(사실을 툭툭 떨구는 건조한 톤)는 여전히 피한다.
독자 옆에서 차근차근 풀어 주는 **AI 어시스턴트 답변체**가 기준이다.
짧은 문장과 긴 문장을 섞어 읽는 리듬을 살린다.

한 단락 예시:
> Anthropic이 27년 묵은 OpenBSD 결함을 찾아냈습니다. 자동화 도구를 500만 번 돌려도 안 잡히던 결함이죠. AI의 취약점 발견 능력이 이미 최정상급 전문가 수준에 올라섰다는 뜻입니다.

격식 — 부드러운 격식 — 격식으로 굴린다.
"옆에서 풀어 주는" 톤은 살되, 글의 무게는 그대로다.

**대제목(`##` 헤드라인) = 짧은 명사형 라벨이 기본.**
그 섹션이 다루는 주제를 짧은 명사구로 적는다.
○ `## 코드 유출 원인` / `## Anthropic의 대응` / `## 에이전트 루프` — 그 글만의 주제어가 들어간 라벨.
○ 서술형(`~했다`/명사 종결)도 짧으면 가능 — `## 1억 달러 투입`, `## 빅테크 11곳과 손잡았다`.
✗ `## 무슨 일인가` / `## 왜 중요한가` / `## 어떻게 동작하나` — 그 글만의 주제어가 없는 빈 목차 라벨.
✗ `~했어요/~예요/~거든요` 같은 구어체 — 헤드라인은 본문보다 한 단계 무겁게 잡는다(본문에서 구어를 섞더라도 헤드라인은 명사구·짧은 서술형으로).
(글 제목은 예외 — 2-0절대로 후킹하게.)

**번역체 금지 — 한국어로 다시 쓴다.**
영어 문장을 한 줄씩 옮기지 않는다.
`~에 가깝습니다`, `~된 생각은 단순합니다`, 무생물 주어 + `~이 있었습니다` 같은 직역투를 걷어낸다.

**한국어 어순으로 — 영어 도치 강조 ✗.**
"흥미로운 건 X입니다 / 절묘한 건 ~입니다 / 중요한 건 ~다" 같은 영어 cleft 강조 도치는 한국어로 풀어 쓴다.
✗ "흥미로운 건 공개 시점이다. ... 시점이 절묘하다."
○ "공식 문서가 나온 시점이 공교롭다. 사람들이 유출 코드를 뜯어보던 무렵, Anthropic이 ..."
강조는 도치 대신 단정 평서문에 부연을 얹는다.
"칼로 자른 듯" 같은 진부한 비유도 피한다.

**번역이 아니라 "읽고 이해한 사람의 설명".**
원문을 한 번 소화한 사람이 한국 독자에게 직접 풀어 주는 톤으로 쓴다.
"문서를 옮긴" 느낌을 내는 다음을 걷어낸다:


- **매 문장 출처 표딱지 ✗.**
  "Anthropic에 따르면"을 문장마다 달지 않는다.
  글 전체가 그 원문 정리임은 맨 위 인용구·말미 출처로 이미 약속돼 있다.
- **단문 나열 ✗ — 맥락으로 잇는다.**
  사실 A·B·C를 토막으로 떨구지 말고, 원문에 이미 있는 인과·배경을 살려 "왜 그런지"를 독자가 따라오게 흐름을 만든다.
- **모를 용어는 한 줄 풀어 준다.**
  약어·전문어는 원문 사실 범위 안에서 가볍게 짚어 준다(가르치듯).
  단, 추측·배경지식 창작 ✗.

여전히 지켜야 할 선: 사실·수치·인용은 정확하게, 지어내지 않는다.
감상은 얼마든지 얹되 사실로 둔갑시키지 않는다 — 무엇이 원문이고 무엇이 내 생각인지 드러나게.

**기술 용어·명령·모드명은 음차하지 않는다 — 영어 그대로.**
`diff` 같은 명령·코드·식별자처럼 개발자가 영어 그대로 쓰는 말은 한글 소리로 옮기지 않는다.
"디프"가 아니라 `diff`로 쓴다.
음차하면 어색하고 검색도 되지 않는다.
영어 그대로 쓰되, 명령·식별자는 `인라인 코드`로 감싸도 좋다.

**제품 UI의 모드·메뉴·플래그 이름도 영어 원표기.**
한국 개발자도 실제로 영어로 부르는 이름은 영어 그대로 적는다.
✗ "기본 모드 / 자동 수락 편집 모드 / 플랜 모드" — 어색한 번역.
○ "**Default** / **Auto-accept edits** / **Plan mode** / **Auto mode**" — 원표기 그대로.
설명만 한국어로 곁들인다.

단, 이미 한국어로 굳어진 외래어(코드·테스트·에이전트·플러그인·리팩터링 등)는 그대로 둔다.

**두괄식 — 두괄은 대제목이 담당.**
대제목이 그 섹션의 결론을 이미 담는다.
본문은 `핵심은 이겁니다 / 규모가 큽니다` 같은 군더더기 도입부 없이 바로 내용으로 들어간다.
글 전체도 맨 위 인용구에서 결론(무엇을·왜)부터 제시한다.
"숨어 있었다 → 알고 보니 ~였다" 식 미괄식 빌드업 ✗.

**요약·문장은 사실부터 직진 — 부사어구·메타 도입 ✗.**
요약 인용구와 각 문단은 **첫 문장이 곧 정보**가 되게 쓴다.
"그런데 ~ 다음 수가 좀 의외였죠", "참 절묘하죠", "따라가 봅니다", "살펴봅시다" 같은 감상·전환·메타 도입은 빼고 사실을 바로 던진다.
✗ "유출 사고부터 동작 원리까지 따라가 봅니다." — 글을 안내하는 메타 문장.
✗ "그런데 Anthropic의 다음 수가 의외였죠 —" — 감상으로 사실 전달을 늦춤.
○ "Anthropic은 사태 직후 공식 문서에 동작 원리를 정리해 공개했습니다." — 사실 직진.
감상·해석은 본문 안에서 사실 다음에 얹는다.

**날짜는 알 수 있는 만큼 정확히 — 사건 글은 일자까지.**
유출·발표·출시 같은 사건은 "2026년 3월"이 아니라 **"2026년 3월 31일"**처럼 일자까지 적는다.
원문에 일자가 있으면 반드시 옮기고, 월 단위로 뭉뚱그리지 않는다.
일자가 정말 없을 때만 월·분기로 좁힌다(추측한 날짜 ✗).

**대제목 = 그 글의 주제어가 들어간 라벨.**
짧은 명사구가 기본, 짧은 서술형도 가능 — 어느 쪽이든 **그 글에서만 나오는 고유 주제어**(고유명사·제품명·핵심 키워드)가 들어가야 한다.
대제목만 위→아래로 훑어도 원문의 전모가 잡히게 한다.
그 아래 단락은 "더 알고 싶은 사람만 펴 읽는" 상세가 되게 한다.

```markdown
> (인용구 = 카드·검색·페이지 두괄식 훅이 같이 쓰는 단일 요약 — 무엇을·왜. -습니다 체)

(강연·발표 영상 글이면 여기에 발표자 소개 문단 — 아래 "발표자 소개" 절 참고)

## (그 섹션의 주제어 라벨 — 짧은 명사구가 기본)
도입부 없이 바로 사실부터. 한 섹션은 한두 단락으로 가볍게("분량과 호흡" 절).

## (다음 주제 라벨 — 발표의 전모/이름)
무엇을·왜. 산문이 기본이되, 스캔되는 사실 묶음(수치·항목 나열)은 불릿으로 떼어 내도 좋다.

## (다음 주제 라벨 — 규모·결과 등 한 토막씩)
한 헤드라인 = 한 메시지. 빠뜨리면 안 되는 사실 위주로 짧게.

## (마지막 라벨 = 핵심·시사점, 또는 글쓴이의 마무리)
원문이 짚는 메시지를 정리하거나, 글을 덮으며 든 생각으로 맺는다. 발언은 발화자 명시.

## 참고 자료
- 원문: [제목 — Anthropic](원문 URL)

> 이 글은 Anthropic 원문을 번역·정리한 것입니다. 원저작권은 Anthropic에 있습니다.
```

> 위 골격은 **한 가지 예시**다. 모든 글이 "요약 → 헤드라인 4개 → 마무리 라벨 → 참고 자료"를 그대로 따를 필요는 없다.
> 글의 길이, 헤드라인 개수, 마무리 절의 유무·위치는 원문 결에 맞게 매번 달라져야 한다.
> 같은 형태가 직전 글과 그대로 겹치면 그게 자동화 신호다 (2-3절).

원칙: **사실은 정확하게 ✓ / 사실 왜곡·창작 ✗.**
요약·재구성·개인 감상은 자유.
출처(원문 링크)는 필수 — `## 참고 자료`에 적는다.
저작권 줄(맨 아래 `>` 인용)은 **기본은 생략**한다.
참고 자료에 출처 링크가 있으면 그것으로 충분하다.
원문이 라이선스를 강하게 요구하거나, 번역·재구성 비중이 커서 출처 강조가 필요한 글에만 예외적으로 단다.

맨 위 `>` 인용구는 `.prose blockquote` 스타일(`src/app/globals.css`)이 잡혀 있어 핵심 요약 박스로 잘 보인다.
이 인용구가 페이지에서 보는 두괄식 훅이자 **카드·검색·OG·RSS의 excerpt 컬럼에도 그대로 들어가는 같은 문장**이다.
결론(무엇을·왜)을 여기서 못 박는다.

### 분량과 호흡 — 헤드라인은 많아도, 한 칸은 가볍게

길이의 문제는 헤드라인 개수가 아니다.
**한 헤드라인 아래에 글이 빽빽하게 들어차는 것**이 문제다.
화면이 글자로 꽉 차면 독자는 읽기도 전에 지친다.
헤드라인은 필요한 만큼 쓰되, 그 아래는 가볍게 비운다.

- **한 헤드라인 = 한 토막.**
  그 섹션은 한 가지만 말한다. 한 단락, 길어도 두 단락.
  단락은 2~4문장. 할 말이 더 있으면 헤드라인을 새로 판다.
- **긴 문단은 무조건 쪼갠다.**
  대여섯 문장이 한 덩어리로 이어지면 벽처럼 보인다.
  짧은 문단·한 줄 문장을 섞어 숨 쉴 틈을 만든다.
- **스캔되는 사실은 불릿(`-`)으로 떼어 낸다.**
  수치 묶음·복수 항목·"이런 것들"이 나열되면 한 문단에 욱여넣지 말고 짧은 불릿으로 시각적 여백을 만든다.
  단, 산문 흐름을 끊는 한두 문장은 그대로 두는 게 낫다 — 모든 단락을 불릿으로 바꾸면 글이 발표 자료처럼 차가워진다.
- **리스트는 동격인 항목 3개 이상일 때만.**
  단일 동작·인과 흐름·"A이고 그래서 B" 같은 자연스러운 산문은 리스트로 바꾸지 않는다.
  ✗ "체크포인트는 ~다.\n- 스냅샷을 떠 둠\n- Esc 두 번으로 복구\n- 단, 외부 영향은 못 막음" — 한 장치의 동작 설명이라 산문이 자연스러움.
  ○ 모드·갈래·항목처럼 **병렬 비교**가 필요한 곳에서만 리스트.
- **다 담지 않는다.**
  원문의 곁가지·반복 설명·부연 예시는 덜어낸다.
  핵심 사실과 결론만 남기고, 나머지는 한 문장으로 압축하거나 버린다.
- **시각자료·헤드라인으로 끊어 준다.**
  설명이 길어지는 곳은 카드로 정리하거나 헤드라인으로 자른다.
  본문과 시각자료가 같은 말을 반복하지는 않는다 — 한쪽이 요약하면 다른 쪽은 짧게.

### 발표자 소개 — 강연·발표 영상 글에 한해

강연·인터뷰·발표 영상을 다루는 글은, 맨 위 `>` 요약 **바로 다음·첫 `##` 헤드라인 앞**에 **발표자 소개 문단**을 둔다.
본문에 들어가기 전에 "이 사람이 누구인가"를 먼저 짚어, 독자에게 호기심을 주고 글에 신빙성을 싣는다.

- 담을 것: 발표자가 **누구인지**(이름·소속), **신뢰를 뒷받침하는 이력·업적**, **지금 하는 일**.
  2~4문장으로 자연스럽게 풀어 준다.
- 독자가 "이 사람 이야기라면 들어볼 만하다"고 느끼게 하되 — **사실만**.
  과장·미사여구·추어올리는 수식 ✗.
- 영상이 발표자를 충분히 소개하지 않으면 WebFetch로 공개된 사실을 확인한다.
  확실하지 않으면 영상이 밝힌 범위로 좁힌다(추측·창작 ✗).
- **첫 문장이 매번 "X는 ~인 사람입니다" 정형으로 시작하지 않게 한다.**
  그 사람의 두드러진 사실(최근 한 일, 화제가 된 이유, 특이한 약력)부터 던지고 정체는 두 번째 문장에서 풀어도 좋다.
  도입 첫 문장 패턴은 매 글마다 의도적으로 다르게 잡는다 (2-3절).
- **발표자가 없는 글**(아티클 번역 등)에는 넣지 않는다.

### GitHub 저장소 소개 글 — 저장소를 다루는 글에 한해

URL-초안 출처가 GitHub 저장소(`github.com/{owner}/{repo}`)면, 그 저장소를 한국 독자에게 소개·정리하는 글이다.
Anthropic 글 번역이 아니라 **저장소의 README와 공개 정보를 풀어 주는 글**이지만, 말투·강조·시각자료 규약은 그대로 적용한다.
요즘 Claude 스킬·AI 도구가 저장소로 많이 공개된다.
"이게 무엇이고 어떻게 쓰는가"를 또렷이 전달하는 것이 목적이다.

근거는 **README와 저장소 페이지에 적힌 내용**이다.
거기 없는 기능·로드맵·수치는 지어내지 않는다(원문 보존 원칙과 같다).

**글 골격**

```markdown
> (요약 = 이 저장소가 무엇이고 왜 흥미로운지 — 무엇을·왜. -습니다 체)

(저장소 소개 문단 — 아래 설명 참고)

## (무엇을 하는 저장소인가 — 서술형 헤드라인)
어떤 문제를 풀고 무엇을 제공하는지. Claude 스킬·AI 도구면 무엇을 자동화·확장
하는지가 중심.

## (설치하고 쓰는 법 — 서술형 헤드라인)
README 기준 설치·실행·구동 방법. 명령은 코드 블록으로.

## Reference
- GitHub Repo: [{owner}/{repo}](저장소 URL)

> 이 글은 {owner}/{repo} 저장소의 README와 공개 정보를 정리해 소개한 글입니다.
```

- **저장소 소개 문단**은 맨 위 `>` 요약 **바로 다음·첫 `##` 앞**에 둔다 — "발표자 소개"와 같은 자리·역할이다.
  누가/어디서 만들었는지(작성자·조직), 얼마나 주목받는지(스타 수), 주 언어·라이선스, 무엇을 위한 저장소인지를 **2~4문장 산문**으로 푼다.
  스타·작성자 같은 메타정보는 별도 표·카드로 빼지 말고 **문장에 자연스럽게 녹인다**.
- **첫 문장이 매번 "{owner}/{repo}는 ~한 저장소입니다" 정형으로 시작하지 않게 한다.**
  저장소의 두드러진 사실(최근 화제, 별난 동작 방식, 만든 사람의 한 줄)부터 던지고 정체는 두 번째 문장에서 풀어도 좋다.
  도입 첫 문장 패턴은 매 글마다 의도적으로 다르게 잡는다 (2-3절).
- `##` 라벨은 가이드일 뿐 — 헤드라인은 "2. 말투" 절대로 그 저장소에 맞는 서술형 문장으로 쓴다.
  동작 원리가 흥미로우면 `## 어떻게 동작하는가` 절을 더해도 된다.
- 설치·실행 명령은 ` ```bash ` 같은 코드 블록에 적는다.
  README에 없는 명령은 만들지 않는다.
  단계가 길면 `step-card`로 흐름을 요약해도 된다(시각자료 한도 유지).
- `## Reference` 절의 헤드라인과 링크 라벨은 영어로 — 섹션 제목은 `Reference`, 저장소 링크는 `GitHub Repo:`, 그 밖의 관련 링크는 `Related:`로 적는다.
- Reference의 저장소 링크는 GitHub 저장소 URL이고, 저작권 표기는 Anthropic이 아니라 저장소·작성자 기준으로 적는다(라이선스를 알면 함께 적어도 좋다).

## 2-1. 강조 표기 · 이모지 (밋밋함 방지)

본문이 헤드라인만 있고 문단은 평문뿐이면 읽기 지친다.
**핵심 단어·수치·발언에 강조를 입혀** 눈이 머물 지점을 만든다.
단, 강조는 **희소할수록 강하다** — 한 문단에 강조가 서넛이면 강조가 아니라 잡음이다.
정말 중요한 곳에만.

| 표기 | 쓸 곳 |
|---|---|
| `**굵게**` | 그 단락의 핵심 주장·결론, 빠뜨리면 안 되는 수치/금액/비율 (구절 단위, 문장 통째 ✗) |
| `*이탤릭*` | 약한 강조, 용어를 처음 꺼낼 때, 미묘한 뉘앙스 |
| `` `인라인 코드` `` | 모델·벤치마크·API·명령·기술 식별자 (예: `Mythos Preview`, `CyberGym`, `SWE-bench`). 일반 회사·제품명은 코드로 감싸지 않는다 |
| `> 인용` | 원문 발언(발화자 명시) 또는 원문이 못 박은 한 덩어리 |
| `<u>밑줄</u>` | 시각 포인트가 꼭 필요한 곳 한정. 드물게 |
| `<mark>형광펜</mark>` | 글 전체에서 가장 중요한 한 구절. **글당 0~1회 · 매번 같은 위치 ✗** (기본은 빈 채로 둔다) |
| `~~취소선~~` | "이제는 아니다" 식 정정·대비 |

- 한 단락에 굵게는 보통 0~1개.
  같은 단어를 매번 굵게 ✗ (첫 등장만).
- **`<mark>` 형광펜은 빈 채로 두는 게 기본.**
  글 전체에 정말 못 박을 한 구절이 있을 때만 1회. 매 글마다 하나씩 넣지 않는다.
- **강조 위치 자체가 정형이 되지 않게 한다.**
  마지막 헤드라인 직전(글 80~90% 지점)에 형광펜이 매번 박히는 패턴은 자동화 티의 대표 사례다.
  넣을 때도 위치(앞·중·끝)는 직전 글과 다르게.
- 인용구·헤드라인 안에는 강조 표기를 넣지 않는다(그 자체가 이미 강조).
- **`**굵게**`는 한글 글자 바로 뒤에서 닫는다.**
  닫는 `**`가 `)`·`%` 같은 문장부호 뒤·한글 앞에 오면 마크다운이 굵게로 읽지 못해 `**`가 화면에 그대로 노출된다.
  `**핵심 개념(Key Concept)**입니다` ✗ → `**핵심 개념**(Key Concept)입니다` ○ — 영문 괄호는 굵게 밖으로 뺀다.
- **이모지(Tossface): 친근하게, 적극적으로 — 단 남발 ✗.**
  이 블로그는 Tossface 컬러 이모지 폰트를 도입했다(폰트 스택 맨 뒤 `"Tossface"`, `src/app/globals.css`).
  딱딱한 평문에 토스페이스 이모지를 곁들여 친근한 톤을 낸다 — 텍스트로 이모지를 쓰면 자동으로 토스 스타일로 렌더된다.
  본문 문장 끝·절 단위에 가볍게, 그리고 ` ```visual ` 카드의 아이콘 자리(`"icon": "💡"`처럼 이모지를 직접)에 쓴다.
  단 장식·도배는 피한다 — 한 문단에 여럿 깔리면 잡음이다. 맨 위 인용구·헤드라인·참고/저작권 줄에는 넣지 않는다.

## 2-2. 시각자료 카탈로그 (이해를 돕는 시각자료)

평문·헤드라인만으로 흐름·구조·비율이 머리에 안 그려지는 부분에 한해 시각자료를 얹는다.
본문에 ` ```visual ` 코드 블록 안에 **JSON**(패턴 + 데이터)만 적으면, 블로그가 그 블록을 **React 카탈로그 컴포넌트로 본문에 그대로 렌더**한다.
색은 디자인 토큰이라 라이트/다크 자동 대응.
이미지로 굽지 않으므로 선명하고 본문 폭에 맞게 반응형으로 표시된다.
블록 안에 HTML/CSS는 쓰지 않는다(디자인 일관성은 카탈로그가 책임진다).

> 카탈로그는 **다이어그램·구조·지표**를 일관되게 그리기 위한 것이다.
> 원문에 이미 있는 그림·사진은 카탈로그 대상이 아니다 — 아래 "작성 규칙" 참고.

**작성 흐름**

````markdown
```visual
{
  "pattern": "step-card",
  "alt": "Claude Code의 에이전트 루프 5단계",
  "title": "Claude Code는 에이전트 루프로 작동합니다",
  "steps": [
    { "label": "사용자 요청",   "sublabel": "your prompt",   "icon": "message-circle" },
    { "label": "컨텍스트 수집", "sublabel": "read · search", "icon": "search" },
````

블로그가 위 블록을 발견하면 해당 카탈로그 컴포넌트를 본문에 렌더한다.
`alt`는 반드시 적는다(접근성·검색 노출에 쓰인다).
디자인 변경은 컴포넌트 쪽에서 한 번 고치면 모든 글의 시각자료에 일관 적용된다.

### 언제 쓰는가

- **직선형 과정·순서·파이프라인** → `step-card` (요청 → 수집 → 실행 → … 한 방향으로 흐르는 단계). 되돌아가 **반복되는 순환**은 step-card로 못 그리니 illustration으로 — 본문이 "반복"을 말하면 시각자료도 반복을 보여야 한다
- **숫자·지표 강조** → `stat-card` (벤치마크 점수, 금액, 점유율 등 1~4개 지표)
- **핵심 인사이트 박스** → `callout-card` (한 문장으로 못 박는 결론·경고·요점)
- **둘·셋 견주기** → `compare-card` (기존 방식 vs 새 방식, A안 vs B안 — 항목을 칼럼으로 나란히)
- **시간 순 흐름** → `timeline-card` (출시 연혁, 사건 순서 — 날짜가 붙는 흐름. 번호 절차는 step-card)
- **원문·강연 인용** → `quote-card` (발표자·원작자가 한 말을 그대로 옮길 때)
- **권장·금지** → `checklist-card` (이렇게 하라 / 이건 피하라 — 체크·금지 표시로 구분)
- **순서 없는 구성요소** → `grid-card` (한 개념을 이루는 갈래·요소를 아이콘 타일로. 순서가 있으면 step-card)
- **짧은 키·설명 나열** → `list-card` (모드 목록, 처리 순서, 키-설명 요약. 단순 `-` 마크다운 리스트보다 한 단계 강조하고 싶을 때)
- **한 줄짜리 짧은 나열** → `bullet-card` (설명 없이 라벨만 — "이런 모습입니다" 류의 압축 목록. 키·설명이 필요하면 list-card)
- **개념·은유 그림** → ` ```illustration ` (구조 스케치, 비유적 그림 — 정확한 수치가 아니라 직관을 그릴 때)

본문에 이미 한 문장으로 충분히 전달된 사실은 시각자료로 또 옮기지 않는다.
**본문이 설명한 구조·결론을 한눈에 보이게 압축**할 때만 쓴다.
시각자료에서 새 사실·추측을 만들어내지 않는다(원문 보존 원칙).

### 공통 옵션

모든 패턴이 공통으로 받는 필드:

| 키 | 타입 | 설명 |
|---|---|---|
| `pattern` | `"step-card" \| "stat-card" \| "callout-card" \| "compare-card" \| "timeline-card" \| "quote-card" \| "checklist-card" \| "grid-card" \| "list-card" \| "bullet-card"` | 카탈로그 패턴 ID. 필수 |
| `alt` | string | 이미지 alt 텍스트. 필수 (접근성·SEO) |
| `eyebrow` | string? | 카드 상단 작은 카테고리/맥락 라벨 (예: `HOW CLAUDE CODE WORKS`). **선택** |
| `title` | string? | 카드 메인 타이틀. **명사형·문어체**로 — `~방식`·`~구조`·`~흐름` 같은 명사 종결, `~합니다`체 ✗. 한 줄·30자 이내. **선택** |
| `accent` | `"primary" \| "accent" \| "info" \| "success" \| "warn" \| "danger" \| "mute"` | 카드 강조 색. 기본 `primary` |

> **`eyebrow`·`title`은 선택이다.**
> 카드 하나로 메시지가 분명하면 둘 다 생략하는 게 깔끔하다 — 본문 문단이 이미 맥락을 잡아 주므로 제목 없는 카드가 오히려 군더더기 없이 낫다.
> 제목을 쓸 땐 반드시 명사형(예: "카탈로그 구성 방식" ○ / "카탈로그는 이렇게 구성됩니다" ✗).
> 라이트·다크는 워커가 두 벌 자동 생성하므로 따로 적지 않는다.

### `accent` — 미리 정의된 7색만 쓴다 (하드코딩 ✗)

색상은 디자인 토큰(`src/app/globals.css`)에 묶여 있어 라이트/다크 자동 대응된다.
**역할마다 색을 달리** 입힌다 — 같은 카드를 모두 같은 색으로 깔지 않는다.

| 클래스 | 색 | 쓸 곳 |
|---|---|---|
| `primary` | 블루 | 기본 처리 단계·메인 흐름 |
| `accent`  | 퍼플 | 보조 분기·대안 경로 |
| `info`    | 티얼 | 외부 입력·소스(사용자, API, 원본 데이터) |
| `success` | 그린 | 최종 결과·출력·성공 |
| `warn`    | 옐로우 | 주의·조건부·인간 검토 |
| `danger`  | 레드 | 실패·차단·에러 경로 |
| `mute`    | 그레이 | 부가 정보·비활성·각주성 노드 |

### 패턴별 명세

**`step-card` — 과정/순서**

| 키 | 타입 | 설명 |
|---|---|---|
| `steps` | `Step[]` | 2~8개. 데스크톱에서 **한 줄로 나란히** 펼쳐진다(카드가 본문보다 넓어짐). 3~6개 권장 |

`Step` 형식:

| 키 | 타입 | 설명 |
|---|---|---|
| `label` | string | 단계 이름 (한 줄, 8자 이내 권장) |
| `sublabel` | string? | 영문 보조 라벨(예: `your prompt`). 한 줄, 16자 이내 |
| `icon` | string | Lucide 아이콘 이름. 아래 화이트리스트 참조 |
| `accent` | string? | 그 단계만 다른 색을 쓰고 싶을 때 (드물게) |

`step-card`는 단계 번호(`01`, `02`…)를 카탈로그가 자동으로 매긴다.
모든 단계 카드는 동일한 스타일로 렌더된다(특정 단계 강조 없음).

**`stat-card` — 숫자 강조**

| 키 | 타입 | 설명 |
|---|---|---|
| `stats` | `Stat[]` | 1~4개. 데스크톱에서 **한 줄로 나란히** 펼쳐진다(카드가 본문보다 넓어짐) |

`Stat` 형식:

| 키 | 타입 | 설명 |
|---|---|---|
| `value` | string | 핵심 숫자/금액/비율 (예: `83.1%`, `1억 달러`, `27년`) |
| `label` | string | 무엇을 가리키는 숫자인지 (한 줄, 16자 이내) |
| `caption` | string? | 출처·기간·맥락 (한 줄, 24자 이내) |
| `icon` | string? | Lucide 아이콘 이름. 있으면 지표 위에 작은 아이콘이 붙어 더 풍부해진다 (선택) |
| `accent` | string? | 그 지표만 다른 색을 쓰고 싶을 때 |
| `delta` | `{ value: string, direction: "up" \| "down" \| "flat" }`? | 전기 대비 증감 표기 (있을 때만) |

**`callout-card` — 핵심 인사이트 박스**

| 키 | 타입 | 설명 |
|---|---|---|
| `icon` | string | Lucide 아이콘 이름 |
| `heading` | string | 한 문장 결론 (40자 이내) |
| `body` | string? | 한두 문장 부연 (120자 이내) |

**`compare-card` — 둘·셋 견주기**

| 키 | 타입 | 설명 |
|---|---|---|
| `columns` | `Column[]` | 2~3개. 나란히 배치되어 비교된다 |

`Column` 형식:

| 키 | 타입 | 설명 |
|---|---|---|
| `label` | string | 칼럼 이름 (예: `에이전트 방식`). 한 줄, 14자 이내 권장 |
| `caption` | string? | 칼럼 한 줄 부연 (20자 이내) |
| `icon` | string? | Lucide 아이콘 이름 (선택) |
| `accent` | string? | 칼럼별 색. 보통 한쪽은 `mute`/`danger`, 다른 쪽은 `primary`/`success` |
| `points` | string[] | 1~6개. 각 항목 한 줄 (35자 이내 권장) |

**`timeline-card` — 시간 순 흐름**

| 키 | 타입 | 설명 |
|---|---|---|
| `events` | `Event[]` | 2~6개. 세로 흐름으로 배치 |

`Event` 형식:

| 키 | 타입 | 설명 |
|---|---|---|
| `date` | string | 날짜/시점 (예: `2024.06`). 한 줄, 10자 이내 권장 |
| `label` | string | 사건 이름 (한 줄, 20자 이내 권장) |
| `description` | string? | 한 줄 부연 (48자 이내) |
| `icon` | string? | Lucide 아이콘 이름. 없으면 점으로 표시된다 |
| `accent` | string? | 그 사건만 다른 색을 쓰고 싶을 때 |

> 번호가 의미 있는 **절차**는 `step-card`, 날짜·연혁 같은 **시점**이 핵심이면 `timeline-card`.

**`quote-card` — 원문·강연 인용**

| 키 | 타입 | 설명 |
|---|---|---|
| `quote` | string | 인용문 한 단락 (110자 이내 권장). **원문 그대로** — 지어내지 않는다. 따옴표는 카탈로그가 자동으로 감싸므로 직접 넣지 않는다 |
| `attribution` | string? | 말한 사람 이름 (20자 이내) |
| `role` | string? | 직함·소속 (24자 이내) |

`quote-card`는 `eyebrow`·`title`을 렌더하지 않는다 — 인용문과 발언자만 보여 준다.
차분한 중립 배경이라 `accent`는 발언자 줄의 작은 강조선에만 쓰인다.

**`checklist-card` — 권장·금지**

| 키 | 타입 | 설명 |
|---|---|---|
| `items` | `Item[]` | 2~8개 |

`Item` 형식:

| 키 | 타입 | 설명 |
|---|---|---|
| `text` | string | 항목 한 줄 (40자 이내 권장) |
| `state` | `"do" \| "dont" \| "neutral"` | 권장(체크·그린)·금지(엑스·레드)·중립(대시·그레이). 기본 `do` |
| `caption` | string? | 항목 한 줄 부연 (40자 이내) |

`checklist-card`의 항목 색은 `state`로 정해진다 — `accent`는 항목 표시에 영향을 주지 않는다.

**`grid-card` — 순서 없는 구성요소**

| 키 | 타입 | 설명 |
|---|---|---|
| `items` | `Item[]` | 2~6개. 데스크톱에서 **한 줄로 나란히** 펼쳐진다(카드가 본문보다 넓어짐). 3~5개 권장 |

`Item` 형식:

| 키 | 타입 | 설명 |
|---|---|---|
| `title` | string | 요소 이름 (한 줄, 14자 이내 권장) |
| `description` | string? | 한두 줄 설명 (48자 이내) |
| `icon` | string | Lucide 아이콘 이름 |
| `accent` | string? | 그 타일만 다른 색을 쓰고 싶을 때 |

> 갈래에 **순서·번호**가 있으면 `step-card`, 그냥 나열되는 **구성요소**면 `grid-card`.

**`list-card` — 키·설명 나열 (모드, 처리 순서, 요약)**

| 키 | 타입 | 설명 |
|---|---|---|
| `items` | `Item[]` | 2~8개. 세로로 나열 |

`Item` 형식:

| 키 | 타입 | 설명 |
|---|---|---|
| `label` | string | 키 — 모드명·단계명·항목명 (한 줄, 24자 이내 권장) |
| `description` | string? | 한 줄 설명 (60자 이내 권장) |

좌측 accent 컬러 바 + 옅은 틴트의 단일 박스로 렌더된다. 아이콘 없음·순서 번호 없음 — 단순한 강조 리스트가 필요할 때 쓴다.
**언제 안 쓰나:** 순서가 핵심이면 `step-card`, 구성요소 갈래면 `grid-card`, 권장/금지면 `checklist-card`. 본문 흐름에 자연스럽게 녹는 짧은 나열은 그냥 `-` 마크다운으로 둔다.

**`bullet-card` — 한 줄 라벨만 짧게 나열**

| 키 | 타입 | 설명 |
|---|---|---|
| `items` | `string[]` | 2~8개. 한 줄·60자 이내. 설명 없이 라벨만 |

`list-card`와 같은 박스 톤이되, 항목은 accent 색 작은 점 + 한 줄 라벨로 가볍게 떨어진다. "구체적으로는 이런 모습입니다" 다음의 압축 목록처럼, **설명을 굳이 붙일 필요 없는 짧은 나열**에 쓴다.
**언제 안 쓰나:** 키마다 한 줄 설명이 붙으면 `list-card`, 권장/금지로 갈리면 `checklist-card`, 순서가 핵심이면 `step-card`. 본문에서 한 문장으로 충분하면 시각자료를 만들지 않는다.

### ` ```illustration ` — 개념 일러스트 (SVG 직접)

다이어그램·카드로는 안 되는 **개념·은유·구조 스케치**(이해를 돕는 그림)는 ` ```illustration ` 블록에 **SVG를 직접** 그린다.
카탈로그(JSON)와 달리 raw SVG를 그대로 적는다 — 워커가 라이트·다크 두 벌 PNG로 굽는 흐름은 동일하다.

글마다 일러스트가 **한 가족처럼** 보이도록 아래 스타일을 반드시 지킨다.

- **스타일: 플랫 라인 일러스트.**
  면은 옅게 채우고 윤곽은 선으로.
  그라데이션·그림자·사실적 묘사·3D ✗.
- **`viewBox="0 0 800 450"` 고정**(16:9). `width`/`height` 속성은 쓰지 않는다.
- **선 두께 2.5**, `stroke-linecap="round"`·`stroke-linejoin="round"`.
- **색은 디자인 토큰을 `style`로 지정한다.**
  `stroke="..."` 같은 프리젠테이션 속성이 아니라 `style="stroke: var(--fg-strong)"` 형태여야 라이트/다크가 자동 대응된다.
  - 윤곽선: `var(--fg-strong)`, 보조선: `var(--fg-neutral)`
  - 면 채움: `var(--diag-blue-fill)` 등 (위 accent 7색 토큰의 `*-fill`)
  - 강조 윤곽: `var(--diag-blue-stroke)` 등 (`*-stroke`)
- 한 일러스트에 **3색 이내** (메인 + accent 1~2개).
- 첫 자식으로 **`<title>설명</title>`** — 접근성 alt로 쓰인다. 필수.
- SVG 안 글자는 최소화한다.
  구조도·순환도라면 노드마다 짧은 라벨은 괜찮지만(수집·실행 등), 설명 문장은 넣지 않는다 — 설명은 본문이 한다.
- `<script>`·외부 참조·`<image>`(래스터) ✗.
- **흐름 방향을 일관되게.**
  좌→우로 흐르면 끝까지 좌→우로 — 폭이 모자라면 박스를 좁혀 맞추지, 흐름을 아래로 꺾지 않는다.
- **여백·간격·겹침을 의식한다.**
  - 좌·우·상·하에 **30px 이상 안전 여백** — 박스 끝이 viewBox 가장자리(800, 450)에 닿으면 렌더 시 잘려 보인다.
  - 박스와 화살표 사이에 **8~10px 간격** — 화살표 꼬리가 박스에 붙으면 답답해 보인다.
  - 라벨 텍스트는 화살표·곡선과 겹치지 않게 **별도 y 좌표 영역**에 둔다 (예: 곡선 apex가 y=150이면 라벨은 y=180 이하).
- 글당 1개. 정말 그림이 이해를 돕는 곳에만.

````markdown
```illustration

  <title>요청 하나가 여러 작업으로 갈라지는 모습</title>
  <circle cx="150" cy="225" r="46"
````

### 아이콘 — Lucide 화이트리스트

아이콘은 [Lucide](https://lucide.dev) 이름을 그대로 적는다(`message-circle`, `search`, `edit`, `check-check`, …).
라이브러리에 없는 이름을 쓰면 워커가 경고하고 기본 점 아이콘으로 폴백한다.
**아래 목록 안에서 고르는 것을 권장**한다 — 카탈로그의 시각적 톤이 일관되게 유지된다.

- 대화·요청: `message-circle`, `message-square`, `send`, `mic`
- 탐색·읽기: `search`, `book-open`, `file-text`, `eye`, `compass`
- 작성·실행: `edit`, `pencil`, `play`, `terminal`, `code`, `rocket`
- 결정·분기: `git-branch`, `split`, `arrow-right-left`, `shuffle`, `scale`, `git-compare`
- 검증·완료: `check`, `check-check`, `circle-check`, `shield-check`, `thumbs-up`
- 경고·실패: `alert-triangle`, `alert-circle`, `x`, `shield-alert`, `ban`
- 데이터·저장: `database`, `hard-drive`, `server`, `cloud`, `archive`
- 사람·조직: `user`, `users`, `building-2`, `briefcase`
- 시간·과정: `clock`, `timer`, `calendar`, `history`, `loader`, `flag`, `milestone`
- 보안·키: `shield`, `lock`, `key`, `fingerprint`

- AI·도구: `bot`, `cpu`, `sparkles`, `zap`, `wrench`, `puzzle`

### 크기 제한 — 작을수록 강하다

- **항목이 3개 이상일 때만 시각자료를 만든다.**
  `step-card`·`stat-card`·`grid-card`·`timeline-card`·`checklist-card`는 나열할 항목이 1~2개면 카드로 키우지 말고 본문 문장으로 적는다 — 2개짜리 카드는 한 줄 문장만 못하다.
  (`callout-card`·`quote-card`는 본래 단일, `compare-card`는 2~3 대상을 견주는 게 목적이라 이 규칙과 무관하다.)
- 한 패턴 안의 요소 수: `step-card` 3~8단계, `stat-card` 3~4지표, `callout-card` 단일, `compare-card` 2~3칼럼, `timeline-card` 3~6사건, `checklist-card` 3~8항목, `grid-card` 3~6타일, `list-card`·`bullet-card` 3~8항목.
- `step-card`·`stat-card`·`grid-card`는 **항목이 한 줄로** 펼쳐진다.
  개수가 많아지면 칸이 한 줄을 지키느라 좁아지고, 모바일처럼 폭이 좁을 때만 여러 줄로 접힌다(에디터 프리뷰에서도 한 줄).
  칸이 너무 좁아 답답하지 않게 **step 3~6 · stat 3~4 · grid 3~5**를 권장한다 — 그보다 많으면 항목을 추리거나 본문 리스트로 옮긴다.
- **글당 0~3개. 기본값은 0.**
  시각자료는 본문 평문만으로 구조·비율·흐름이 그려지지 않을 때만 얹는다.
  "수치가 있으니 한 개는 만들어야지"가 아니라 "이 부분은 카드 없이는 안 잡힌다"가 만드는 이유여야 한다.
  **시각자료가 0개인 글이 절반은 나와도 정상**이다.
- **매번 같은 패턴 조합·같은 위치 금지.**
  직전 글이 `stat-card` + `compare-card` 두 개를 본문 30%·60% 지점에 깔았다면, 이번 글은 시각자료를 빼거나, 다른 패턴(`callout-card`·`timeline-card`·`step-card`·`grid-card` 등)을 다른 지점에 한 개만 쓴다.
  **`stat-card` + `compare-card` 2개 조합 반복이 가장 명백한 자동화 티다.** (2-3절)
- 시각자료 다 채우기보다 **중심 메시지 하나**를 또렷이 — 한 시각자료 = 한 결론.

### 작성 규칙

- ` ```visual ` 블록 안에는 **JSON만** 적는다.
  주석·후행 콤마 ✗ (`JSON.parse`에서 깨진다).
- 라벨·캡션은 **한 줄, 글자 수 권장 안에서.**
  길어지면 본문 문단으로 옮긴다(시각자료는 구조만).
- `accent`는 위 7개 외 값 ✗. 직접 hex/HTML 색 지정 ✗.
- 본문이 이미 설명한 사실을 **요약·구조화**할 때만 쓴다.
  시각자료에서 새 사실·추측을 만들어내지 않는다(원문 보존 원칙).
- 시각자료에 별도 캡션·각주를 달지 않는다.
  바로 위·아래 문단이 맥락을 잡는다.
  필요한 부연은 `eyebrow`·`alt`로 충분하다.
- **원문의 그림·사진·도표는 적극 활용한다.**
  원문에 이해를 돕는 시각자료가 있으면 카탈로그로 새로 그리지 말고 `![설명](원문 이미지 URL)` 마크다운 이미지로 본문에 가져온다.
  원문 이미지 URL을 직접 링크해도 된다.
- **YouTube 글은 소제목별로 영상 장면을 캡처해 넣을 수 있다.**
  `npm run capture`(절차는 `docs/draft-in-claude-code.md`)로 각 소제목 시점의 후보 프레임을 뽑아, 세션이 직접 보고 가장 나은 컷 1장을 고른다.
  슬라이드·자료·핵심 화면처럼 **본문 내용을 뒷받침하는 장면**만 넣는다 — 흐림·전환 중·눈 감음·빈 화면·발표자 표정만 잡힌 컷은 배제하고, 쓸 만한 게 없으면 그 소제목은 건너뛴다.
  캡처도 시각자료 한도·변주(2-3절) 안에서 — 매 글 같은 자리·개수로 반복하지 않는다.
- 카탈로그·이미지로 안 되는 형태는 본문 표·리스트, 또는 직접 HTML/CSS로 표현해도 된다.
  다만 같은 패턴이 반복되면 카탈로그 컴포넌트로 추가하는 편이 디자인 일관성에 낫다(`src/components/visuals/`).

## 2-3. 자동화 티 방지 — 글 사이 변주

이 블로그의 초안은 Claude Code 세션이 직접 쓴다.
규약이 상한선만 정해 두면 매 글이 상한선 근처에서 비슷한 형태로 수렴한다.
글 몇 개만 훑어도 "같은 손이 같은 틀로 찍어 냈다"는 인상이 남는다.
이를 피하려면 **개별 글의 규약 준수**보다 **글과 글 사이의 다양성**을 한 단계 위에서 본다고 생각하라.

초안을 쓰기 전에 `drafts/` 안의 최근 글 1~2개를 훑어 다음 항목이 직전 글과 겹치지 않는지 확인한다.

- **종결 비율.**
  매 문장 `~습니다.`로 끊지 말고, `~죠/~까요/~네요/~거든요` 같은 양념 종결이 한 글에 몇 군데 자연스럽게 박혔는지 본다.
  반대로 양념이 한 문단에 둘 이상 깔렸으면 가볍다 — 한 글당 2~3개 선에서 줄인다.
- **글 길이.**
  본문 줄 수가 직전 글의 ±15% 안에 들면 의식적으로 늘리거나 줄인다.
- **헤드라인 개수와 호흡.**
  매번 `##` 4개 + 마무리 라벨로 떨어지지 않게 한다. 세 개로 짧게 끝내도 되고, 다섯 개로 길게 가도 된다.
- **도입 첫 문장.**
  "X는 ~입니다" 정형이 두 글 연속이면 이번에는 사실·장면·인용·수치로 시작한다.
- **`<mark>` 위치.**
  직전 글의 형광펜이 글 끝쪽이었다면 이번 글은 형광펜을 빼거나 앞쪽에 둔다.
- **시각자료 개수·패턴 조합.**
  직전 글이 2개였다면 이번은 0~1개.
  직전 조합이 `stat-card` + `compare-card`였다면 이번은 그 조합을 피하고 덜 쓰인 패턴(`grid-card`·`list-card`·`timeline-card`·`quote-card`·`callout-card`)을 시도한다.
- **마무리 저작권 줄.**
  직전 글에 박혔다면 이번 글은 생략한다(기본은 생략).

원칙은 단순하다 — **직전 글과 같은 구조가 보이면 일부러 다른 길을 고른다.**
변주는 규약을 어기는 것이 아니라 규약의 일부다.
