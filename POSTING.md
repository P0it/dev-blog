## 1. 에디터 입력 — velog식 미니멀

`/admin/editor`(`src/components/admin/PostEditor.tsx`)는 velog처럼 **제목·태그·카테고리·본문**
넷만 입력하고, 나머지는 전부 자동이다.

| 필드 | 어떻게 채워지나 |
|---|---|
| 제목 | 상단 큰 입력칸에 직접. 원제 직역 금지·한국어 독자용 재작성(무엇을/왜가 드러나게) |
| 태그 | 제목 아래 입력칸에 쉼표 구분 (`anthropic, 보안, ai-security` 등) |
| 카테고리 | 작은 드롭다운 (`anthropic` 계열 / 주제 카테고리) |
| 본문 | 마크다운. 이미지는 **드래그·붙여넣기**로 자동 업로드(별도 버튼 없음) |
| ~~슬러그~~ | **자동** — 제목에서 slugify. 기존 글은 URL 안정성을 위해 원래 슬러그 유지 |
| ~~요약(excerpt)~~ | **자동** — 본문 첫 `>` 블록쿼트에서 파생(`deriveExcerpt`). 카드/검색/SNS/SEO/RSS/OG가 보는 문장과 페이지상 두괄식 훅이 **같은 문장**이다 |
| ~~썸네일 종류~~ | **자동** — 슬러그 해시로 a–f 6변형에 결정적 분포(`thumbKindFromSlug`). 카드 단조로움 방지 |
| ~~읽는 시간~~ | **자동** — 본문 분량에서 산출(`deriveReadingMin`) |
| ~~시리즈 · 추천 옵션~~ | 입력란 없음. 기존 값은 row update에 포함 안 해 보존됨 |

> 본문은 `> 요약` → `## 헤드라인` 순서로 시작한다(맨 위에 `# 제목`을 다시 쓰지 않는다 —
> 제목은 입력칸이 권위). 본문 첫 `>` 인용구는 페이지의 두괄식 훅이면서 동시에
> `excerpt` 컬럼에 자동 추출되어 카드·OG·RSS에 동일 문장으로 들어간다.

## 2. 말투 · 본문 골격

**목적: 정보 전달(가공·전달).**
원문을 새로 해석하거나 의견을 덧대지 않는다. 원문의 사실·논지·수치·인용을
누락 없이, 원문이 강조한 비중 그대로 한국어로 옮기는 것이 목표다.


**원문 최대한 보존.** 핵심 주장·숫자·고유명사·발언(인용)을 원문에 충실하게.
선택적으로 추려도 의미·강조점을 바꾸지 않는다. 인용은 발화자를 원문대로 명시.

**말투: 본문은 `-습니다/-입니다` 체.** 해요체·`~거든요/~죠/~예요` 같은
구어 종결은 쓰지 않는다. 다만 **신문 기사 톤은 피한다** — 사실을 툭툭
떨구는 건조한 보도체가 아니라, 독자 옆에서 차근차근 풀어 주는 **따뜻한
설명체**로 쓴다. 종결은 -습니다로 단정하게, 문장은 설명하듯 부드럽게.
짧은 문장과 긴 문장을 섞어 읽는 리듬을 살린다.

**제목(대제목)은 문어체.** `~했어요/~예요` 같은 구어체 ✗.
`~했다` 또는 명사형 종결(예: "…1억 달러 투입")의 헤드라인 문어체로.

**번역체 금지 — 한국어로 다시 쓴다.** 영어 문장을 한 줄씩 옮기지 않는다.
`~에 가깝습니다`, `~된 생각은 단순합니다`, 무생물 주어 + `~이 있었습니다`
같은 직역투를 걷어낸다.

**번역이 아니라 "읽고 이해한 사람의 설명".** 원문을 한 번 소화한 사람이 한국
독자에게 직접 풀어 주는 톤으로 쓴다. "문서를 옮긴" 느낌을 내는 다음을 걷어낸다:


- **매 문장 출처 표딱지 ✗.** "Anthropic에 따르면"을 문장마다 달지 않는다. 글
  전체가 그 원문 정리임은 맨 위 인용구·말미 출처로 이미 약속돼 있다.
- **단문 나열 ✗ — 맥락으로 잇는다.** 사실 A·B·C를 토막으로 떨구지 말고, 원문에
  이미 있는 인과·배경을 살려 "왜 그런지"를 독자가 따라오게 흐름을 만든다.
- **모를 용어는 한 줄 풀어 준다.** 약어·전문어는 원문 사실 범위 안에서 가볍게
  짚어 준다(가르치듯). 단, 추측·배경지식 창작 ✗.

여전히: 사실·수치·인용 보존, 새 해석·사견·"내 생각" 섹션 ✗. 바뀌는 건
**전달 목소리**(옮긴이 → 이해한 설명자)이지, 내용의 출처·범위가 아니다.

**두괄식 — 두괄은 대제목이 담당.** 대제목이 그 섹션의 결론을 이미 담으므로,
본문은 `핵심은 이겁니다 / 규모가 큽니다` 같은 군더더기 도입부 없이 바로
내용으로 들어간다. 글 전체도 맨 위 인용구에서 결론(무엇을·왜)부터 제시한다.
"숨어 있었다 → 알고 보니 ~였다" 식 미괄식 빌드업 ✗.

**대제목 = 그 자체로 정보가 되는 서술형 문장.**
`## 무슨 일인가` / `## 왜 중요한가` 같은 목차형 라벨은 쓰지 않는다.
대제목만 위→아래로 훑어도 원문의 전모가 잡히고, 그 아래 단락은
"더 알고 싶은 사람만 펴 읽는" 상세가 되게 한다.

```markdown
> (인용구 = 카드·검색·페이지 두괄식 훅이 같이 쓰는 단일 요약 — 무엇을·왜. -습니다 체)

## (핵심 메시지 ①을 담은 문어체 헤드라인 — `~했다`/명사형)
도입부 없이 바로 사실부터. 원문 근거대로 서술.

## (핵심 메시지 ②를 담은 문어체 헤드라인 — 발표의 전모/이름)
무엇을·왜. 숫자·파트너·금액을 문장 안에 자연스럽게 녹인다 (불릿 ✗).

## (핵심 메시지 ③… — 규모·결과 등 한 토막씩 헤드라인으로)
한 헤드라인 = 한 메시지. 원문에서 빠뜨리면 안 되는 사실 위주.

## (마지막 헤드라인 = 원문이 짚는 핵심/시사점)
원문이 강조한 메시지를 정리. 발언은 발화자 명시. 개인 논평 ✗.

## 참고 자료
- 원문: [제목 — Anthropic](원문 URL)

> 이 글은 Anthropic 원문을 번역·정리한 것입니다. 원저작권은 Anthropic에 있습니다.
```

원칙: **충실한 번역·정리 ✓ / 개인 창작·논평 ✗.** 출처(원문 링크)와
저작권 표기는 필수. 가공 = 한국어 번역 + 한국 독자용 재구성(두괄식·헤드라인)까지만.

맨 위 `>` 인용구는 `.prose blockquote` 스타일(`src/app/globals.css`)이 잡혀 있어 핵심 요약 박스로 잘 보인다.
이 인용구가 페이지에서 보는 두괄식 훅이자 **카드·검색·OG·RSS의 excerpt 컬럼에도 그대로 들어가는 같은 문장**이다 — 결론(무엇을·왜)을 여기서 못 박는다.

## 2-1. 강조 표기 · 이모지 (밋밋함 방지)

본문이 헤드라인만 있고 문단은 평문뿐이면 읽기 지친다. **핵심 단어·수치·발언에
강조를 입혀** 눈이 머물 지점을 만든다. 단, 강조는 **희소할수록 강하다** — 한 문단에
강조가 서넛이면 강조가 아니라 잡음이다. 정말 중요한 곳에만.

| 표기 | 쓸 곳 |
|---|---|
| `**굵게**` | 그 단락의 핵심 주장·결론, 빠뜨리면 안 되는 수치/금액/비율 (구절 단위, 문장 통째 ✗) |
| `*이탤릭*` | 약한 강조, 용어를 처음 꺼낼 때, 미묘한 뉘앙스 |
| `` `인라인 코드` `` | 모델·벤치마크·API·명령·기술 식별자 (예: `Mythos Preview`, `CyberGym`, `SWE-bench`). 일반 회사·제품명은 코드로 감싸지 않는다 |
| `> 인용` | 원문 발언(발화자 명시) 또는 원문이 못 박은 한 덩어리 |
| `<u>밑줄</u>` | 시각 포인트가 꼭 필요한 곳 한정. 드물게 |
| `<mark>형광펜</mark>` | 글 전체에서 가장 중요한 한 구절. **글당 0~2회** |
| `~~취소선~~` | "이제는 아니다" 식 정정·대비 |

- 한 단락에 굵게는 보통 0~1개. 같은 단어를 매번 굵게 ✗ (첫 등장만).
- 인용구·헤드라인 안에는 강조 표기를 넣지 않는다(그 자체가 이미 강조).
- **이모지: 최소화.** 정돈된 설명체라 본문엔 대개 불필요하다.
  가독성·구분에 실질적으로 도움이 될 때만, 섹션당 최대 1개 선에서. 장식·남발 ✗.
  맨 위 인용구·헤드라인·참고/저작권 줄에는 쓰지 않는다.

## 2-2. 시각자료 카탈로그 (이해를 돕는 시각자료)

평문·헤드라인만으로 흐름·구조·비율이 머리에 안 그려지는 부분에 한해 시각자료를
얹는다. 본문에는 ` ```visual ` 코드 블록 안에 **JSON**을 적어 두고, 워커가
후처리 단계에서 React 카탈로그 컴포넌트로 **라이트·다크 두 벌의 PNG**를 굽고
본문 이미지로 치환한다(`scripts/lib/post-process-visuals.mjs`). ` ```visual `
블록 안에는 **패턴(`pattern`) + 데이터**(JSON)만 적는다(블록 안에 HTML/CSS ✗)
— 디자인 일관성은 카탈로그가 책임진다.

> 카탈로그는 **다이어그램·구조·지표**를 일관되게 그리기 위한 것이다. 원문에
> 이미 있는 그림·사진은 카탈로그 대상이 아니다 — 아래 "작성 규칙" 참고.

**작성 흐름**

````markdown
```visual
{
  "pattern": "step-card",
  "alt": "Claude Code의 에이전트 루프 5단계",
  "title": "Claude Code는 에이전트 루프로 작동합니다",
  "steps": [
    { "label": "사용자 요청",   "sublabel": "your prompt",   "icon": "message-circle" },
    { "label": "컨텍스트 수집", "sublabel": "read · search", "icon": "search" },
    { "label": "실행",         "sublabel": "edit · run",    "icon": "edit" },
    { "label": "결과 검증",     "sublabel": "test · verify", "icon": "check" },
    { "label": "작업 완료",     "sublabel": "task complete","icon": "check-check", "highlight": true }
  ]
}
```
````

워커가 위 블록을 발견하면 라이트·다크 PNG를 각각 굽고, 본문을 `<figure>`로
치환한다. 블로그 테마에 따라 둘 중 하나만 표시된다(`globals.css`).

`alt`는 반드시 적는다(접근성·SNS·검색 노출에 쓰인다). 디자인 변경은 컴포넌트
쪽에서 한 번 고치면 모든 글의 시각자료에 일관 적용된다.

### 언제 쓰는가

- **과정·순서·파이프라인** → `step-card` (사용자 요청 → 컨텍스트 수집 → 실행 → … 같은 단계)
- **숫자·지표 강조** → `stat-card` (벤치마크 점수, 금액, 점유율 등 1~4개 지표)
- **핵심 인사이트 박스** → `callout-card` (한 문장으로 못 박는 결론·경고·요점)

본문에 이미 한 문장으로 충분히 전달된 사실은 시각자료로 또 옮기지 않는다. **본문이
설명한 구조·결론을 한눈에 보이게 압축**할 때만 쓴다. 시각자료에서 새 사실·추측을
만들어내지 않는다(원문 보존 원칙).

### 공통 옵션

모든 패턴이 공통으로 받는 필드:

| 키 | 타입 | 설명 |
|---|---|---|
| `pattern` | `"step-card" \| "stat-card" \| "callout-card"` | 카탈로그 패턴 ID. 필수 |
| `alt` | string | 이미지 alt 텍스트. 필수 (접근성·SEO) |
| `eyebrow` | string? | 카드 상단 작은 카테고리/맥락 라벨 (예: `HOW CLAUDE CODE WORKS`). **선택** |
| `title` | string? | 카드 메인 타이틀. **명사형·문어체**로 — `~방식`·`~구조`·`~흐름` 같은 명사 종결, `~합니다`체 ✗. 한 줄·30자 이내. **선택** |
| `accent` | `"primary" \| "accent" \| "info" \| "success" \| "warn" \| "danger" \| "mute"` | 카드 강조 색. 기본 `primary` |

> **`eyebrow`·`title`은 선택이다.** 카드 하나로 메시지가 분명하면 둘 다 생략하는
> 게 깔끔하다 — 본문 문단이 이미 맥락을 잡아 주므로 제목 없는 카드가 오히려 군더더기
> 없이 낫다. 제목을 쓸 땐 반드시 명사형(예: "카탈로그 구성 방식" ○ / "카탈로그는
> 이렇게 구성됩니다" ✗). 라이트·다크는 워커가 두 벌 자동 생성하므로 따로 적지 않는다.

### `accent` — 미리 정의된 7색만 쓴다 (하드코딩 ✗)

색상은 디자인 토큰(`src/app/globals.css`)에 묶여 있어 라이트/다크 자동 대응된다.
**역할마다 색을 달리** 입힌다 — 같은 카드를 모두 같은 색으로 깔지 않는다.

| 클래스 | 색 | 쓸 곳 |
|---|---|---|
| `primary` | 블루 | 기본 처리 단계·메인 흐름 |
| `accent`  | 퍼플 | 보조 분기·대안 경로 |
| `info`    | 티얼 | 외부 입력·소스(사용자, API, 원본 데이터) |
| `success` | 그린 | 최종 결과·출력·성공 |
| `warn`    | 옐로우 | 주의·조건부·인간 검토 |
| `danger`  | 레드 | 실패·차단·에러 경로 |
| `mute`    | 그레이 | 부가 정보·비활성·각주성 노드 |

### 패턴별 명세

**`step-card` — 과정/순서**

| 키 | 타입 | 설명 |
|---|---|---|
| `steps` | `Step[]` | 2~8개. 길이로 자동 배치 (4개까지 가로, 5개부터 자동 줄바꿈 또는 세로 권장) |
| `direction` | `"horizontal" \| "vertical"` | 기본 `horizontal` |

`Step` 형식:

| 키 | 타입 | 설명 |
|---|---|---|
| `label` | string | 단계 이름 (한 줄, 8자 이내 권장) |
| `sublabel` | string? | 영문 보조 라벨(예: `your prompt`). 한 줄, 16자 이내 |
| `icon` | string | Lucide 아이콘 이름. 아래 화이트리스트 참조 |
| `highlight` | boolean? | `true`면 그 단계만 강조 색·테두리로 부각. 카드당 0~1개 |
| `accent` | string? | 그 단계만 다른 색을 쓰고 싶을 때 (드물게) |

`step-card`는 단계 번호(`01`, `02`…)와 단계 사이 화살표를 카탈로그가 자동으로
그린다. 본문에서 번호·화살표를 적지 않는다.

**`stat-card` — 숫자 강조**

| 키 | 타입 | 설명 |
|---|---|---|
| `stats` | `Stat[]` | 1~4개. 길이로 자동 배치 |

`Stat` 형식:

| 키 | 타입 | 설명 |
|---|---|---|
| `value` | string | 핵심 숫자/금액/비율 (예: `83.1%`, `1억 달러`, `27년`) |
| `label` | string | 무엇을 가리키는 숫자인지 (한 줄, 16자 이내) |
| `caption` | string? | 출처·기간·맥락 (한 줄, 24자 이내) |
| `icon` | string? | Lucide 아이콘 이름. 있으면 지표 위에 작은 아이콘이 붙어 더 풍부해진다 (선택) |
| `accent` | string? | 그 지표만 다른 색을 쓰고 싶을 때 |
| `delta` | `{ value: string, direction: "up" \| "down" \| "flat" }`? | 전기 대비 증감 표기 (있을 때만) |

**`callout-card` — 핵심 인사이트 박스**

| 키 | 타입 | 설명 |
|---|---|---|
| `icon` | string | Lucide 아이콘 이름 |
| `heading` | string | 한 문장 결론 (40자 이내) |
| `body` | string? | 한두 문장 부연 (120자 이내) |

### 아이콘 — Lucide 화이트리스트

아이콘은 [Lucide](https://lucide.dev) 이름을 그대로 적는다(`message-circle`,
`search`, `edit`, `check-check`, …). 라이브러리에 없는 이름을 쓰면 워커가
경고하고 기본 점 아이콘으로 폴백한다. **아래 목록 안에서 고르는 것을 권장**한다 —
카탈로그의 시각적 톤이 일관되게 유지된다.

- 대화·요청: `message-circle`, `message-square`, `send`, `mic`
- 탐색·읽기: `search`, `book-open`, `file-text`, `eye`, `compass`
- 작성·실행: `edit`, `pencil`, `play`, `terminal`, `code`, `rocket`
- 결정·분기: `git-branch`, `split`, `arrow-right-left`, `shuffle`
- 검증·완료: `check`, `check-check`, `circle-check`, `shield-check`, `thumbs-up`
- 경고·실패: `alert-triangle`, `alert-circle`, `x`, `shield-alert`, `ban`
- 데이터·저장: `database`, `hard-drive`, `server`, `cloud`, `archive`
- 사람·조직: `user`, `users`, `building-2`, `briefcase`
- 시간·과정: `clock`, `timer`, `calendar`, `history`, `loader`
- 보안·키: `shield`, `lock`, `key`, `fingerprint`
- 측정·지표: `bar-chart-3`, `trending-up`, `trending-down`, `gauge`, `activity`
- 금융·숫자: `dollar-sign`, `coins`, `wallet`, `receipt`
- AI·도구: `bot`, `cpu`, `sparkles`, `zap`, `wrench`, `puzzle`

### 크기 제한 — 작을수록 강하다

- 한 패턴 안의 요소 수: `step-card` 2~8단계, `stat-card` 1~4지표, `callout-card`는 단일.
- 글당 **보통 1개, 많아야 2개.** 강조와 같은 원칙 — 흔하면 무의미하다.
- 시각자료 다 채우기보다 **중심 메시지 하나**를 또렷이 — 한 시각자료 = 한 결론.

### 작성 규칙

- ` ```visual ` 블록 안에는 **JSON만** 적는다. 주석·후행 콤마 ✗ (`JSON.parse`에서 깨진다).
- 라벨·캡션은 **한 줄, 글자 수 권장 안에서.** 길어지면 본문 문단으로 옮긴다(시각자료는 구조만).
- `accent`는 위 7개 외 값 ✗. 직접 hex/HTML 색 지정 ✗.
- `highlight: true`는 카드당 0~1개 (첨부 이미지의 첫 번째 또는 마지막 단계처럼).
- 본문이 이미 설명한 사실을 **요약·구조화**할 때만 쓴다. 시각자료에서 새 사실·
  추측을 만들어내지 않는다(원문 보존 원칙).
- 시각자료에 별도 캡션·각주를 달지 않는다. 바로 위·아래 문단이 맥락을 잡는다.
  필요한 부연은 `eyebrow`·`alt`로 충분하다.
- **원문의 그림·사진·도표는 적극 활용한다.** 원문에 이해를 돕는 시각자료가
  있으면 카탈로그로 새로 그리지 말고 `![설명](원문 이미지 URL)` 마크다운
  이미지로 본문에 가져온다. 원문 이미지 URL을 직접 링크해도 된다.
- 카탈로그·이미지로 안 되는 형태는 본문 표·리스트, 또는 직접 HTML/CSS로
  표현해도 된다. 다만 같은 패턴이 반복되면 카탈로그 컴포넌트로 추가하는
  편이 디자인 일관성에 낫다(`src/components/visuals/`).

## AI가 27년 묵은 보안 결함을 찾아냈다

사람이 **27년** 동안 못 찾은 OpenBSD 결함을 Anthropic의 미공개 모델 `Mythos Preview`가 찾아냈습니다. FFmpeg에 16년간 박혀 있던 취약점도 마찬가지였는데, *자동화 도구를 500만 번 돌려도* 걸러지지 않던 결함입니다. 사람·기존 도구가 수십 년 놓친 걸 잡아냈다는 건, AI의 취약점 발견 능력이 이미 **최정상급 보안 전문가 수준**에 올라섰다는 뜻입니다.

## Anthropic이 11개 기업과 함께 출범시킨 Project Glasswing

AWS, 애플, 브로드컴, 시스코, 크라우드스트라이크, 구글, JP모건체이스, 리눅스 재단, 마이크로소프트, 엔비디아, 팔로알토 네트웍스까지 11개 기업·기관이 참여했습니다. 취지는 분명합니다. <mark>공격자가 먼저 손에 넣기 전에 방어자에게 닿게 한다</mark> — AI의 취약점 발견 능력이 이미 임계점을 넘었다는 판단입니다.

## 찾아낸 취약점은 수천 개, 모두 보고·패치를 마쳤다

`Mythos Preview`는 주요 운영체제와 웹 브라우저 전반에서 고위험 취약점을 수천 개 찾아냈고, 리눅스 커널에서는 권한 상승으로 이어지는 취약점을 스스로 엮어내기까지 했습니다. 보안 벤치마크 `CyberGym`에서는 **83.1%**를 기록해 기존 `Claude Opus 4.6`의 66.6%를 크게 앞섰고, `SWE-bench` 등 일반 코딩·추론 벤치마크에서도 이전 세대를 큰 폭으로 넘어섰습니다. 발견된 결함은 모두 보고되어 패치를 마친 상태입니다.

## Anthropic, 방어 보안에 1억 달러 투입

방어 보안 작업에 쓸 모델 사용 크레딧 **1억 달러**와, 오픈소스 보안 단체(Alpha-Omega·OpenSSF·아파치 재단 등)에 대한 **400만 달러** 직접 기부를 약속했습니다. 또한 90일 뒤에는 그동안 학습한 취약점과 수정·개선 내용을 공개하겠다고 밝혔습니다.

> 취약점 발견에서 악용까지 걸리던 시간이 몇 달에서 몇 분으로 무너졌습니다.
