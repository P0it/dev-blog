# 포스팅 작성 가이드 — Anthropic 글 번역·요약

주 워크플로우: [anthropic.com](https://www.anthropic.com/) 의 post / story / announcement / research →
**번역 → 요약 → 본인 해석** 후 발행.

> 코드 변경 없는 **작성 컨벤션** 문서다. 출처/참고자료는 별도 기능이 아니라
> 본문 markdown 안에 약속된 형식으로 적는다. (`Post` 타입의 본문은 `bodyMd` 단일 필드 —
> `src/lib/types.ts`, 렌더링은 `src/components/post/MarkdownView.tsx`, `react-markdown + remark-gfm`)

## 1. 에디터 메타 패널 매핑

`/admin/editor` 의 좌측 메타 패널(`src/components/admin/PostEditor.tsx`) 필드를 이렇게 채운다.

| 필드 | 채우는 법 |
|---|---|
| 제목 | 원제 직역 금지. 한국어 독자용으로 재작성 (무엇을/왜가 드러나게) |
| 슬러그 | 원문 URL의 마지막 경로 활용 (예: `glasswing` → `project-glasswing`) |
| 요약(excerpt) | 2~3문장. "무엇을 발표했고, 왜 중요한가". **목록·검색·SNS·SEO 전용 카탈로그 문구** — 글 상세 페이지에는 노출되지 않는다 |
| 카테고리 | `anthropic` 계열 / 주제 카테고리 |
| 태그 | `anthropic` + 주제 태그 (`보안`, `ai-security` 등) |
| ~~읽는 시간~~ | **자동 계산** — 본문 분량에서 산출(`deriveReadingMin`, `src/lib/markdown.ts`). 더는 손으로 입력하지 않는다 |

> ⚠️ **요약(excerpt) ↔ 본문 맨 위 `>` 인용구 = 베껴 쓰지 않는다.**
> 둘 다 "무엇을·왜"를 다루지만 역할이 다르다. **excerpt**는 글을 아직 안 연 사람을 위한
> 카드/검색 결과용 카탈로그 문구(상세 페이지엔 안 보임). **본문 맨 위 `>` 인용구**는 글을
> 연 사람이 화면에서 처음 보는 **유일한 요약 박스**(두괄식 훅). 같은 사실을 담더라도
> 문장·도입 각도를 달리해 **복붙·동어반복이 되지 않게** 쓴다.

## 2. 말투 · 본문 골격

**목적: 정보 전달(가공·전달). 개인 창작·논평이 아니다.**
원문을 새로 해석하거나 의견을 덧대지 않는다. 원문의 사실·논지·수치·인용을
누락 없이, 원문이 강조한 비중 그대로 한국어로 옮기는 것이 목표다.
"그래서 내 생각은" 류의 논평 섹션을 만들지 않는다.

**원문 최대한 보존.** 핵심 주장·숫자·고유명사·발언(인용)을 원문에 충실하게.
선택적으로 추려도 의미·강조점을 바꾸지 않는다. 인용은 발화자를 원문대로 명시.

**말투: 본문은 `-습니다/-입니다` 체.** 해요체·`~거든요/~죠/~예요` 같은
구어 종결을 쓰지 않는다. 차분한 정보 전달 톤.

**제목(대제목)은 문어체.** `~했어요/~예요` 같은 구어체 ✗.
`~했다` 또는 명사형 종결(예: "…1억 달러 투입")의 헤드라인 문어체로.

**번역체 금지 — 한국어로 다시 쓴다.** 영어 문장을 한 줄씩 옮기지 않는다.
`~에 가깝습니다`, `~된 생각은 단순합니다`, 무생물 주어 + `~이 있었습니다`
같은 직역투를 걷어낸다.

**번역이 아니라 "읽고 이해한 사람의 설명".** 원문을 한 번 소화한 사람이 한국
독자에게 직접 풀어 주는 톤으로 쓴다. "문서를 옮긴" 느낌을 내는 다음을 걷어낸다:

- **거리두기 술어 남발 ✗.** 일반 사실까지 `원문은 ~라고 한다 / ~라고 설명한다 /
  ~라고 밝혔다`로 감싸 옮긴이 말투를 내지 않는다. 이해한 내용은 글쓴이가 **직접
  단언**한다. (단, 특정 인물·기관의 *발언·입장*은 발화자를 명시 — 그건 유지.)
- **매 문장 출처 표딱지 ✗.** "Anthropic에 따르면"을 문장마다 달지 않는다. 글
  전체가 그 원문 정리임은 맨 위 인용구·말미 출처로 이미 약속돼 있다.
- **단문 나열 ✗ — 맥락으로 잇는다.** 사실 A·B·C를 토막으로 떨구지 말고, 원문에
  이미 있는 인과·배경을 살려 "왜 그런지"를 독자가 따라오게 흐름을 만든다.
- **모를 용어는 한 줄 풀어 준다.** 약어·전문어는 원문 사실 범위 안에서 가볍게
  짚어 준다(가르치듯). 단, 추측·배경지식 창작 ✗.

여전히: 사실·수치·인용 보존, 새 해석·사견·"내 생각" 섹션 ✗. 바뀌는 건
**전달 목소리**(옮긴이 → 이해한 설명자)이지, 내용의 출처·범위가 아니다.

**두괄식 — 두괄은 대제목이 담당.** 대제목이 그 섹션의 결론을 이미 담으므로,
본문은 `핵심은 이겁니다 / 규모가 큽니다` 같은 군더더기 도입부 없이 바로
내용으로 들어간다. 글 전체도 맨 위 인용구에서 결론(무엇을·왜)부터 제시한다.
"숨어 있었다 → 알고 보니 ~였다" 식 미괄식 빌드업 ✗.

**대제목 = 그 자체로 정보가 되는 서술형 문장.**
`## 무슨 일인가` / `## 왜 중요한가` 같은 목차형 라벨은 쓰지 않는다.
대제목만 위→아래로 훑어도 원문의 전모가 잡히고, 그 아래 단락은
"더 알고 싶은 사람만 펴 읽는" 상세가 되게 한다.

```markdown
> (인용구: 원문의 결론 한 덩어리 — 무엇을·왜. -습니다 체)

## (핵심 메시지 ①을 담은 문어체 헤드라인 — `~했다`/명사형)
도입부 없이 바로 사실부터. 원문 근거대로 서술.

## (핵심 메시지 ②를 담은 문어체 헤드라인 — 발표의 전모/이름)
무엇을·왜. 숫자·파트너·금액을 문장 안에 자연스럽게 녹인다 (불릿 ✗).

## (핵심 메시지 ③… — 규모·결과 등 한 토막씩 헤드라인으로)
한 헤드라인 = 한 메시지. 원문에서 빠뜨리면 안 되는 사실 위주.

## (마지막 헤드라인 = 원문이 짚는 핵심/시사점)
원문이 강조한 메시지를 정리. 발언은 발화자 명시. 개인 논평 ✗.

## 참고 자료
- 원문: [제목 — Anthropic](원문 URL)

> 이 글은 Anthropic 원문을 번역·정리한 것입니다. 원저작권은 Anthropic에 있습니다.
```

원칙: **충실한 번역·정리 ✓ / 개인 창작·논평 ✗.** 출처(원문 링크)와
저작권 표기는 필수. 가공 = 한국어 번역 + 한국 독자용 재구성(두괄식·헤드라인)까지만.

맨 위 `>` 인용구는 `.prose blockquote` 스타일(`src/app/globals.css`)이 잡혀 있어 핵심 요약 박스로 잘 보인다.
상세 페이지에서 excerpt는 노출되지 않으므로, **이 인용구가 독자가 보는 유일한 요약**이다 — 결론(무엇을·왜)을 여기서 못 박는다.

## 2-1. 강조 표기 · 이모지 (밋밋함 방지)

본문이 헤드라인만 있고 문단은 평문뿐이면 읽기 지친다. **핵심 단어·수치·발언에
강조를 입혀** 눈이 머물 지점을 만든다. 단, 강조는 **희소할수록 강하다** — 한 문단에
강조가 서넛이면 강조가 아니라 잡음이다. 정말 중요한 곳에만.

| 표기 | 쓸 곳 |
|---|---|
| `**굵게**` | 그 단락의 핵심 주장·결론, 빠뜨리면 안 되는 수치/금액/비율 (구절 단위, 문장 통째 ✗) |
| `*이탤릭*` | 약한 강조, 용어를 처음 꺼낼 때, 미묘한 뉘앙스 |
| `` `인라인 코드` `` | 모델·벤치마크·API·명령·기술 식별자 (예: `Mythos Preview`, `CyberGym`, `SWE-bench`). 일반 회사·제품명은 코드로 감싸지 않는다 |
| `> 인용` | 원문 발언(발화자 명시) 또는 원문이 못 박은 한 덩어리 |
| `<u>밑줄</u>` | 시각 포인트가 꼭 필요한 곳 한정. 드물게 |
| `<mark>형광펜</mark>` | 글 전체에서 가장 중요한 한 구절. **글당 0~2회** |
| `~~취소선~~` | "이제는 아니다" 식 정정·대비 |

- 한 단락에 굵게는 보통 0~1개. 같은 단어를 매번 굵게 ✗ (첫 등장만).
- 인용구·헤드라인 안에는 강조 표기를 넣지 않는다(그 자체가 이미 강조).
- **이모지: 최소화.** 차분한 정보 전달 톤이라 본문엔 대개 불필요하다.
  가독성·구분에 실질적으로 도움이 될 때만, 섹션당 최대 1개 선에서. 장식·남발 ✗.
  맨 위 인용구·헤드라인·참고/저작권 줄에는 쓰지 않는다.

## 3. 출처/참고자료 컨벤션

### 방식 A — 참고 자료 섹션 (기본 권장)

```markdown
## 참고 자료

- 원문: [Project Glasswing — Anthropic](https://www.anthropic.com/glasswing)
- 관련: [관련 글 제목](URL)

> 이 글은 Anthropic의 원문을 번역·요약한 것입니다. 원저작권은 Anthropic에 있습니다.
```

### 방식 B — 각주 (본문 중간 인용 시)

```markdown
Mythos Preview는 CyberGym에서 83.1%를 기록했다[^1].

[^1]: [Project Glasswing, Anthropic](https://www.anthropic.com/glasswing)
```

> ⚠️ 현재 `.prose ul`만 스타일이 있고 `.prose ol` / 각주(`.footnotes`) CSS는 없다
> (`src/app/globals.css`). **방식 A(불릿)는 그대로 깔끔하게 렌더링**되고,
> 방식 B(각주)는 동작은 하나 번호·구분선이 밋밋하다. 번역·요약 글은 **방식 A를 기본**으로.

## 4. 워크드 예시 — `https://www.anthropic.com/glasswing`

**메타:**
- 제목: `프로젝트 글래스윙 — AI가 수십 년 묵은 보안 취약점을 찾아내기 시작했다`
- 슬러그: `project-glasswing`
- 요약: `Anthropic이 11개 기술 기업과 함께 AI를 방어적 보안에 투입하는 Project Glasswing을 발표했다. 미공개 모델 Mythos Preview가 모든 주요 OS·브라우저에서 수천 개의 고위험 취약점을 찾아냈고, 일부는 수십 년간 발견되지 않았던 것이다.`
- 태그: `anthropic, 보안, ai-security, glasswing`
- 읽는 시간: 자동 계산(입력란 없음)

> 위 **요약(excerpt)**과 아래 **본문 맨 위 `>` 인용구**를 비교해 보면, 같은 사실을 다루되
> 도입 각도·문장이 다르다(excerpt=발표 사실의 카탈로그 / 본문 인용구=훅). 복붙이 아니다.
> 본문은 평문 위에 굵게·`코드`·인용·`<mark>`을 **드물게** 얹어 핵심에 시선을 잡는다.
> 또한 사실마다 "~라고 설명합니다"로 감싸지 않고 **이해한 내용을 직접 풀어 주는
> 목소리**다 — 옮긴 글이 아니라 읽고 소화한 사람이 짚어 주는 글로 읽혀야 한다.

**본문:**

```markdown
> Anthropic이 아직 공개하지 않은 AI 모델로 수십 년 묵은 보안 취약점 **수천 개**를 찾아냈습니다. 그리고 이 능력을 혼자 갖지 않고 빅테크 11곳과 함께 방어에 쓰기로 했습니다. 이름은 Project Glasswing입니다.

## AI가 27년 묵은 보안 결함을 찾아냈다

사람이 **27년** 동안 못 찾은 OpenBSD 결함을 Anthropic의 미공개 모델 `Mythos Preview`가 찾아냈습니다. FFmpeg에 16년간 박혀 있던 취약점도 마찬가지였는데, *자동화 도구를 500만 번 돌려도* 걸러지지 않던 결함입니다. 사람·기존 도구가 수십 년 놓친 걸 잡아냈다는 건, AI의 취약점 발견 능력이 이미 **최정상급 보안 전문가 수준**에 올라섰다는 뜻입니다.

## Anthropic이 11개 기업과 함께 출범시킨 Project Glasswing

AWS, 애플, 브로드컴, 시스코, 크라우드스트라이크, 구글, JP모건체이스, 리눅스 재단, 마이크로소프트, 엔비디아, 팔로알토 네트웍스까지 11개 기업·기관이 참여했습니다. 취지는 분명합니다. <mark>공격자가 먼저 손에 넣기 전에 방어자에게 닿게 한다</mark> — AI의 취약점 발견 능력이 이미 임계점을 넘었다는 판단입니다.

## 찾아낸 취약점은 수천 개, 모두 보고·패치를 마쳤다

`Mythos Preview`는 주요 운영체제와 웹 브라우저 전반에서 고위험 취약점을 수천 개 찾아냈고, 리눅스 커널에서는 권한 상승으로 이어지는 취약점을 스스로 엮어내기까지 했습니다. 보안 벤치마크 `CyberGym`에서는 **83.1%**를 기록해 기존 `Claude Opus 4.6`의 66.6%를 크게 앞섰고, `SWE-bench` 등 일반 코딩·추론 벤치마크에서도 이전 세대를 큰 폭으로 넘어섰습니다. 발견된 결함은 모두 보고되어 패치를 마친 상태입니다.

## Anthropic, 방어 보안에 1억 달러 투입

방어 보안 작업에 쓸 모델 사용 크레딧 **1억 달러**와, 오픈소스 보안 단체(Alpha-Omega·OpenSSF·아파치 재단 등)에 대한 **400만 달러** 직접 기부를 약속했습니다. 또한 90일 뒤에는 그동안 학습한 취약점과 수정·개선 내용을 공개하겠다고 밝혔습니다.

## 핵심은 "공격자보다 방어자가 먼저"다

크라우드스트라이크 CTO는 다음과 같이 말합니다.

> 취약점 발견에서 악용까지 걸리던 시간이 몇 달에서 몇 분으로 무너졌습니다.

AI가 공격과 방어 양쪽에 같은 능력을 쥐어준 상황에서, Anthropic은 그 능력을 **방어자에게 먼저, 더 빨리** 닿게 하는 데 무게를 둡니다. 리눅스 재단은 그간 보안 인력이 부족했던 오픈소스 메인테이너에게도 같은 수준의 전문성이 분배된다는 점을 의의로 꼽았습니다.
```
